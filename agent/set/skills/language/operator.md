# Language: operator audience

Write documentation for the operator, not the developer.

- Focus on what to do, not how it works internally.
- Name commands and outcomes, not implementation details.
- Skip jargon when a plain word works: "restart" not
  "bounce the service", "storage disk" not "block device".
- When jargon is necessary, define it on first use.
- Error messages tell the operator what went wrong and
  what to do next -- not stack traces or internal state.
